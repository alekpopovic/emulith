# Task 29 — Emulith v0.2 hardening and release readiness

Target agent: Codex with GPT-5.6  
Prerequisite: Tasks 00–28.

Perform the final hardening, migration, documentation, and release-readiness pass for Emulith `v0.2.0`.

Do not publish, tag, or push anything in this task.

## Required workflow

1. Run the full baseline suite.
2. Inspect architecture, migrations, protocol handlers, generated compatibility data, Docker packaging, release configuration, and docs.
3. Produce a concise prioritized findings list.
4. Fix every release-blocking defect within scope.
5. Add regression tests for every fixed defect.
6. Re-run all acceptance gates.
7. Finish with an explicit `GO` or `NO-GO` recommendation and objective reasons.

Do not stop after the audit.

## Intended v0.2 service scope

The release may include only the tested subset of:

```text
STS
S3
SQS
DynamoDB
SNS
CloudWatch Logs
```

No Azure, GCP, Lambda, dashboard, Kubernetes, or production mode.

## Upgrade and migration testing

Create migration fixtures or programmatic state representing the last `v0.1.0-poc` schema.

Verify:

- opening old state migrates automatically;
- S3/SQS/STS behavior remains intact;
- old objects/messages remain readable;
- migration is idempotent;
- a failed migration leaves a recoverable backup or clear failure state;
- no downgrade is attempted;
- current state export/import succeeds;
- importing an old supported snapshot migrates safely;
- newer unsupported snapshot/schema is rejected;
- migration documentation lists irreversible changes.

Do not fabricate a v0.1 fixture that does not match the actual historical schema in the repository.

## Cross-service scenarios

Add or strengthen end-to-end tests:

### SNS to SQS

- topic/queue/subscription;
- envelope and raw delivery;
- restart persistence;
- deletion cleanup;
- reset.

### Application workflow

A realistic local flow:

1. create DynamoDB table;
2. put an item;
3. publish an SNS event;
4. receive it through SQS;
5. write a CloudWatch Logs event;
6. store an artifact in S3;
7. restart Emulith;
8. verify durable state;
9. export;
10. reset;
11. import;
12. verify the workflow state again.

Use real AWS SDK for Go v2 clients for public operations.

## Concurrency and race hardening

Run and fix:

- `go test -race ./...`;
- concurrent S3 overwrites/deletes;
- SQS competing receivers;
- DynamoDB conditional writers;
- SNS publish while unsubscribe/delete occurs;
- CloudWatch Logs concurrent writers;
- reset/export/import while requests are active;
- graceful shutdown during in-flight requests.

Verify there are no deadlocks, leaked goroutines, unclosed bodies/files/rows, or corrupted state.

## Protocol robustness and fuzzing

Ensure bounded fuzz/unit coverage for:

- AWS target classification;
- query/form parsing;
- AttributeValue decoding;
- DynamoDB expressions;
- pagination tokens;
- snapshot archive parser;
- S3 key/path handling;
- SNS envelope handling;
- CloudWatch filter subset.

Normal CI must run seed corpora quickly. Add a documented longer fuzz command for maintainers.

## Resource limits

Audit and document configurable/default limits for:

- request body;
- S3 object;
- SQS message;
- DynamoDB item/expression/batch;
- SNS message;
- CloudWatch log event/batch;
- snapshot import;
- pagination page size.

Return provider-shaped validation errors, not process crashes or unbounded allocation.

## Compatibility truthfulness

Run generated compatibility checks and verify:

- every `supported` operation has a passing real SDK test;
- every partial behavior lists deviations;
- no stale test IDs;
- no manual edits are overwritten unexpectedly;
- SDK module versions appear in reports;
- README summary matches generated details.

Downgrade a status rather than overstating behavior.

## Security and local threat model

Verify:

- no path/symlink traversal;
- no archive traversal/bombs;
- no credentials/body logging;
- no telemetry/phone home;
- no real-cloud fallback;
- admin endpoints are clearly local/trusted-network only;
- container runs non-root;
- volume permissions work;
- no secrets in workflows/examples;
- least-privilege Actions permissions;
- dependency/license review has no unexplained incompatible code.

Generate/update SBOM and vulnerability scan configuration if present, but do not claim zero vulnerabilities without evidence.

## Performance sanity

Add non-brittle benchmarks or scripts and inspect:

- startup time;
- idle memory;
- thousands of small S3 objects;
- thousands of SQS messages;
- a few thousand DynamoDB items with Query/Scan;
- SNS fan-out to several queues;
- CloudWatch event ingestion/read.

Use results to catch obvious quadratic behavior. Do not turn wall-clock numbers into fragile CI gates.

## Acceptance gate

Run every applicable command:

```bash
gofmt -w <changed-go-files>
go test ./...
go test -race ./...
go vet ./...
make build
make compatibility
make compatibility-report
make compatibility-check
make demo
make docker-build
make release-snapshot
make release-check
```

When Docker is available:

- start the image;
- verify health and non-root UID;
- run S3/SQS/STS/DynamoDB/SNS/Logs SDK flows;
- restart with the same volume;
- verify persistence;
- stop and clean up.
