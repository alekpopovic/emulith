# Governance

Maintainers review changes, releases, security fixes, and compatibility claims. Contributors propose issues, code, tests, and documentation. Decisions are discussed publicly in issues or pull requests except confidential vulnerability handling.

Changes normally proceed through an issue/PR, review, passing CI, and maintainer merge. All commits contributed through PRs require DCO sign-off. Compatibility status is approved only when public provider-shaped tests and real SDK compatibility tests support the claim.

Releases are proposed publicly, checked against the acceptance suite and changelog, then tagged by a maintainer. Maintainers are added or removed through a documented public decision based on sustained contribution, judgment, and conduct. During the small-project phase, existing maintainers reach consensus where practical and record disagreements.

Security fixes may be developed privately until coordinated disclosure. No company or maintainer is entitled to private core correctness features. Governance changes are public. This document describes project practice; it does not override the license or contributor rights.
